# PlotPulse 架构重构设计文档

## 一、重构背景

### 1.1 原架构问题

通过分析《AI的觉醒》小说及原架构代码，发现以下核心问题：

| 问题 | 影响 | 根本原因 |
|:---|:---|:---|
| 缺乏全局故事状态管理 | 情节断裂、角色跳跃 | 每章独立生成，无状态共享 |
| 章节间记忆机制薄弱 | 连贯性差、伏笔丢失 | 只传递500字前情提要 |
| 大纲与执行脱节 | 故事未完成 | 大纲是非结构化文本 |
| 角色状态追踪缺失 | 角色动机不清 | 只有静态设定 |
| 缺乏一致性检查 | 逻辑错误、时间混乱 | 只在最后审阅 |
| 编辑功能简单 | 风格不一致 | 只有语言润色 |
| 流水线模式 | 整体架构问题 | 不适合小说创作 |

### 1.2 重构目标

1. **完整性**: 确保故事结构完整，有明确的开头、发展、高潮、结局
2. **连贯性**: 章节间逻辑连贯，角色状态一致，伏笔得到回收
3. **深度**: 支持情节重构，不仅是表面润色
4. **可控性**: 人类在关键决策点深度参与

### 2.2 核心组件

#### 2.2.1 StoryBrain (中央状态管理器)

**职责**:
- 维护完整的故事状态
- 管理角色状态机
- 追踪情节点和伏笔
- 检查一致性
- 为Agent提供上下文

**核心数据结构**:

```python
class StoryBrain:
    # 角色管理
    characters: Dict[str, Dict]                    # 静态设定
    character_states: Dict[int, Dict[str, CharacterState]]  # 动态状态
    
    # 情节管理
    plot_points: Dict[str, PlotPoint]              # 所有情节点
    active_plot_points: Set[str]                   # 活跃情节点
    resolved_plot_points: Set[str]                 # 已解决情节点
    
    # 伏笔管理
    foreshadowing_planted: Dict[str, Dict]         # 已埋下的伏笔
    foreshadowing_resolved: Dict[str, int]         # 已回收的伏笔
    
    # 章节管理
    chapter_plans: Dict[int, ChapterPlan]          # 章节规划
    chapter_executions: Dict[int, ChapterExecution] # 章节执行结果
```

**关键方法**:
- `check_consistency()`: 检查新章节与已有内容的一致性
- `get_context_for_chapter_planning()`: 为章节规划提供完整上下文
- `get_context_for_chapter_writing()`: 为章节写作提供完整上下文
- `rollback_to_chapter()`: 回滚到指定章节重新创作

#### 2.2.2 Collaborative Agents (多Agent协作系统)

**Agent角色**:

| Agent | 职责 | 解决的原问题 |
|:---|:---|:---|
| PlotDesigner | 设计整体结构、规划章节弧线 | 大纲与执行脱节 |
| CharacterDeveloper | 发展角色弧线、验证行为合理性 | 角色状态追踪缺失 |
| ScenePlanner | 设计具体场景结构、管理场景过渡 | 缺乏场景规划 |
| Writer | 根据规划执行具体写作 | 写作缺乏约束 |
| ConsistencyChecker | 检查跨章节一致性 | 缺乏一致性检查 |
| Editor | 语言润色、结构调整 | 编辑功能简单 |
| QualityAssessor | 评估质量、识别改进空间 | 缺乏质量评估 |

**协作机制**:
- 所有Agent共享StoryBrain状态
- Agent通过`generate_proposal()`生成提案
- Agent通过`evaluate_proposal()`评估其他Agent的提案
- 冲突通过协商机制解决

#### 2.2.3 Iterative Pipeline (迭代式创作流程)

**流程步骤**:

```
设计整体结构
    │
    ▼
逐章创作:
    │
    ├──▶ 规划章节 (PlotDesigner)
    │       └── 确定目标、情节点、角色弧线、伏笔
    │
    ├──▶ 规划场景 (ScenePlanner)
    │       └── 设计场景序列、节奏控制
    │
    ├──▶ 发展角色 (CharacterDeveloper)
    │       └── 确定角色本章状态
    │
    ├──▶ 写作 (Writer)
    │       └── 在约束条件下创作
    │
    ├──▶ 一致性检查 (ConsistencyChecker)
    │       └── 验证与已有内容的连贯性
    │
    ├──▶ 编辑 (Editor)
    │       └── 润色和结构调整
    │
    ├──▶ 质量评估 (QualityAssessor)
    │       └── 评分和提供改进建议
    │
    └──▶ 人类审核
            ├──▶ 批准 → 确认章节
            ├──▶ 修改 → 根据反馈重写
            ├──▶ 回滚 → 回到任意阶段
            └──▶ 中止 → 停止创作
```

**关键特性**:
- 每章经过完整流程，确保质量
- 发现问题可回滚到任意阶段
- 支持深度编辑（重构情节、调整角色）
- 人类在关键决策点深度参与

## 三、关键设计决策

### 3.1 为什么使用中央状态管理器？

**对比**:

| 原架构 | 新架构 |
|:---|:---|
| 每章独立生成 | StoryBrain维护全局状态 |
| 信息通过参数传递 | 信息通过中央存储共享 |
| 难以追踪跨章节信息 | 所有信息可追溯 |
| 无法检查一致性 | 自动一致性检查 |

**好处**:
1. 单一事实来源，避免信息不一致
2. 支持复杂查询（角色弧线、伏笔状态等）
3. 便于实现回滚和重构
4. 支持长期规划和全局优化

### 3.2 为什么需要多Agent协作？

**对比**:

| 原架构 | 新架构 |
|:---|:---|
| 单一WritingAgent | 多个专业Agent |
| 一个Agent负责所有 | 每个Agent负责一个领域 |
| 难以深入每个方面 | 每个方面都有专家处理 |

**好处**:
1. 专业化：每个Agent深入一个领域
2. 可扩展：容易添加新Agent
3. 可验证：Agent间相互检查
4. 透明：每个决策都有明确的责任Agent

### 3.3 为什么使用迭代式而非流水线？

**对比**:

| 流水线模式 | 迭代式模式 |
|:---|:---|
| 单向流动 | 支持回滚 |
| 发现问题只能继续 | 发现问题可以重做 |
| 人类只在最后审核 | 人类在每个关键决策点参与 |
| 浅层编辑 | 深度重构 |

**好处**:
1. 质量保障：问题在早期发现和解决
2. 灵活性：支持各种创作策略
3. 可控性：人类深度参与
4. 可回溯：支持实验和探索

## 五、实施建议

### 5.1 迁移策略

**阶段1: 并行运行**
- 保留原架构
- 新架构作为实验性功能
- 对比两种架构的输出质量

**阶段2: 逐步替换**
- 先用新架构替换规划阶段
- 再替换写作阶段
- 最后完全替换

**阶段3: 完全迁移**
- 删除原架构代码
- 新架构成为默认

### 5.2 性能考虑

**潜在问题**:
- 每章经过多个Agent，调用次数增加
- StoryBrain状态持久化开销
- 一致性检查的计算成本

**优化策略**:
- Agent结果缓存
- 异步并行处理
- 一致性检查抽样

### 5.3 人类参与设计

**关键决策点**:
1. 整体结构设计完成后
2. 每章规划完成后
3. 每章写作完成后
4. 一致性检查发现问题时
5. 质量评估不达标时

**参与方式**:
- GUI界面展示完整信息
- 提供修改建议输入
- 支持回滚到任意阶段
- 实时预览修改效果

## 七、总结

新架构通过以下三个核心设计解决了原架构的根本问题：

1. **StoryBrain**: 提供中央状态管理，确保全局一致性
2. **Collaborative Agents**: 专业化分工，深入处理每个方面
3. **Iterative Pipeline**: 支持迭代优化和深度编辑

这套架构不仅能解决《AI的觉醒》的问题，还能为长期的小说创作提供坚实的基础。
